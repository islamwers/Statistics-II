# ⚙️ Kubernetes

## Что такое Kubernetes

**Kubernetes** — это открытое программное обеспечение для оркестрации контейнеризированных приложений.

Оркестрация включает:

- развёртывание контейнеров;
- перезапуск контейнеров при сбоях;
- обновление приложений;
- масштабирование;
- остановку приложений;
- контроль работоспособности сервисов.

---

## Что делает Kubernetes

Kubernetes помогает автоматически управлять состоянием контейнеров и приложений.

Основные функции:

- автоматическое развёртывание;
- мониторинг сервисов;
- балансировка сетевого трафика;
- оркестрация хранилища;
- самоконтроль;
- перезапуск отказавших контейнеров;
- замена неработоспособных контейнеров;
- поддержание заданного состояния приложения.

---

# 🧱 Архитектура Kubernetes

Kubernetes-кластер состоит из управляющих компонентов и рабочих узлов.

---

## Master / Control Plane

**Master** управляет worker-узлами, распределяет контейнеры, управляет рабочими нагрузками и следит за тем, чтобы приложение и кластер находились в нужном состоянии.

---

## Etcd

**Etcd** — хранилище состояния кластера.

В нём хранится информация:

- о конфигурациях;
- о компонентах кластера;
- о текущем состоянии объектов;
- о желаемом состоянии объектов.

---

## API Server

**API Server** — основная точка управления кластером.

Через API Server пользователи и инструменты конфигурируют:

- deployments;
- roles;
- workloads;
- services;
- другие объекты Kubernetes.

---

## Scheduler

**Scheduler** распределяет рабочую нагрузку по узлам кластера.

Он решает, на каком узле должен быть запущен конкретный pod.

---

## Nodes

**Node** — это машина в Kubernetes-кластере.

На узлах запускаются контейнеры приложения.

---

## Pods

**Pod** — минимальная единица запуска в Kubernetes.

Pod может содержать один или несколько контейнеров, которые запускаются как единое целое и используют общие сетевые и файловые ресурсы.

---

## Kubelet

**Kubelet** — агент, запущенный на worker-нодах.

Он управляет pod-ами, контейнерами, образами и связанными ресурсами на узле.

---

## Proxy

**Proxy** отвечает за сетевую доступность и предсказуемое взаимодействие внутри кластера.

---

## Kubectl

**Kubectl** — интерфейс командной строки для управления Kubernetes-кластером.

Через `kubectl` можно:

- применять манифесты;
- смотреть список pod-ов;
- масштабировать приложения;
- получать информацию о deployment;
- управлять сервисами и другими объектами Kubernetes.

---

# 🟡 Managed Service for Kubernetes

## Что такое Managed Kubernetes

**Managed Service for Kubernetes** — управляемый сервис Yandex Cloud для создания и эксплуатации Kubernetes-кластеров.

Он снижает объём ручной работы по администрированию Kubernetes.

Yandex Cloud берёт на себя часть задач:

- развёртывание виртуальных машин;
- настройку сети;
- установку ОС и ПО;
- обновление кластера и групп узлов;
- доступность master-узла;
- резервное копирование;
- безопасность хранилища данных и оборудования;
- интеграцию с сервисами Yandex Cloud.

# 🚀 Практическая часть 1 — Создание Kubernetes-кластера

В практической части создаётся Kubernetes-кластер и группа узлов в нём.

---

## Шаг 1. Создание кластера

В консоли Yandex Cloud необходимо выбрать сервис:

```text
Managed Service for Kubernetes
```

Затем нажать:

```text
Создать кластер
```

---

## Шаг 2. Сервисные аккаунты

Для Kubernetes нужны сервисные аккаунты:

- сервисный аккаунт для ресурсов;
- сервисный аккаунт для узлов.

Сервисный аккаунт для ресурсов используется для выделения ресурсов в облаке.

Сервисный аккаунт для узлов нужен самим узлам Kubernetes для доступа к другим ресурсам, например к Container Registry.

---

## Шаг 3. Ключ шифрования

Yandex Key Management Service позволяет защищать конфиденциальные данные:

- пароли;
- OAuth-токены;
- SSH-ключи.

В рамках практической работы создание ключа не является обязательным.

---

## Шаг 4. Релизный канал

В Kubernetes доступны релизные каналы:

- `RAPID`;
- `REGULAR`;
- `STABLE`.

Для практической работы выбирается:

```text
REGULAR
```

---

## Шаг 5. Конфигурация мастера

При создании кластера необходимо настроить:

- версию Kubernetes;
- публичный IP-адрес;
- тип мастера;
- зону доступности;
- облачную сеть;
- подсеть.

Тип мастера влияет на отказоустойчивость:

| Тип мастера | Особенности |
|---|---|
| Зональный | Работает в одной зоне доступности |
| Региональный | Работает в трёх зонах доступности |

---

## Шаг 6. Сетевые настройки

Для подов и сервисов Kubernetes автоматически назначаются внутренние IP-адреса.

Важно следить, чтобы адресные диапазоны Kubernetes не пересекались с другими диапазонами в облаке.

CIDR можно оставить пустым — тогда адреса будут назначены автоматически.

---

## Шаг 7. Создание кластера

После заполнения настроек нужно нажать:

```text
Создать кластер
```

Далее необходимо дождаться статуса:

```text
RUNNING
```

и состояния:

```text
HEALTHY
```

# 📦 Практическая часть 4 — Развёртывание приложения

В кластере разворачивается веб-сервер **NGINX**.

Для этого создаётся YAML-манифест:

```text
my-nginx.yaml
```

---

## Основные элементы манифеста

В манифесте используются следующие поля:

| Поле | Назначение |
|---|---|
| `apiVersion` | Версия API Kubernetes |
| `kind` | Тип объекта Kubernetes |
| `metadata` | Метаданные объекта |
| `name` | Имя объекта |
| `spec` | Описание желаемого состояния |
| `replicas` | Количество pod-ов |
| `selector` | Правило выбора pod-ов |
| `template` | Шаблон pod-а |
| `containers` | Описание контейнеров |
| `image` | Docker-образ контейнера |

---

## Пример Deployment-манифеста

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-nginx-deployment
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: cr.yandex/<идентификатор_реестра>/ubuntu-nginx:latest
```

---

## Применение манифеста

```bash
kubectl apply -f <путь_к_файлу_my-nginx.yaml>
```

После успешного применения появится сообщение:

```text
deployment.apps/my-nginx-deployment created
```

---

## Проверка pod-ов

Список pod-ов:

```bash
kubectl get pods
```

Подробная информация:

```bash
kubectl get pods -o wide
```

Максимально подробная информация о deployment:

```bash
kubectl describe deployment/my-nginx-deployment
```

# 📈 Практическая часть 5 — Масштабирование приложения

Количество pod-ов можно увеличить двумя способами.

---

## Способ 1. Через манифест

Изменить значение:

```yaml
replicas: 3
```

и снова выполнить:

```bash
kubectl apply -f <путь_к_файлу_my-nginx.yaml>
```

---

## Способ 2. Через команду scale

```bash
kubectl scale --replicas=3 deployment/my-nginx-deployment
```

После этого можно проверить pod-ы:

```bash
kubectl get pods
```

# ⚖️ Практическая часть 6 — Балансировка нагрузки

После развёртывания приложения внутри кластера к нему ещё нет доступа из интернета.

Чтобы открыть приложение наружу, используется сервис типа:

```text
LoadBalancer
```

---

## Пример манифеста load-balancer.yaml

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-loadbalancer
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
  type: LoadBalancer
```

---

## Основные параметры

| Параметр | Назначение |
|---|---|
| `port` | Порт балансировщика |
| `targetPort` | Порт контейнера |
| `selector` | Метка pod-ов, на которые направляется трафик |
| `type: LoadBalancer` | Создание внешнего балансировщика |

---

## Применение манифеста

```bash
kubectl apply -f <путь_к_файлу_load-balancer.yaml>
```

После успешного применения появится сообщение:

```text
service/my-loadbalancer created
```

В Yandex Cloud автоматически появится балансировщик нагрузки с префиксом `k8s`.

---

## Проверка доступа

Нужно скопировать внешний IP-адрес балансировщика и открыть его в браузере.

Ожидаемый результат:

```text
Welcome to nginx
```

# 🔁 Практическая часть 7 — Автомасштабирование

В Managed Kubernetes рассматриваются два инструмента автомасштабирования:

- **Horizontal Pod Autoscaler**
- **Cluster Autoscaler**

---

## Horizontal Pod Autoscaler

**Horizontal Pod Autoscaler (HPA)** масштабирует pod-ы.

Он увеличивает или уменьшает количество pod-ов в зависимости от нагрузки.

HPA ориентируется на:

- requests;
- limits;
- загрузку CPU;
- целевой уровень утилизации.

---

## Requests и Limits

`requests` задают минимальный объём ресурсов, который нужен pod-у для запуска.

`limits` ограничивают максимальное количество ресурсов, которое может использовать контейнер.

Пример:

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "500m"
  limits:
    memory: "500Mi"
    cpu: "1"
```

---

## Cluster Autoscaler

**Cluster Autoscaler** управляет количеством узлов в Kubernetes-кластере.

Он создаёт новые узлы, если pod-ам не хватает ресурсов, и удаляет лишние узлы, если нагрузка снизилась.

В Yandex Managed Service for Kubernetes `Cluster Autoscaler` включён по умолчанию.

---

## Пример HPA-манифеста

```yaml
apiVersion: autoscaling/v1
kind: HorizontalPodAutoscaler
metadata:
  name: my-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-nginx-deployment-hpa
  minReplicas: 1
  maxReplicas: 5
  targetCPUUtilizationPercentage: 20
```

---

## Применение манифеста автомасштабирования

```bash
kubectl apply -f <путь_к_load-balancer-hpa.yaml>
```

После применения создаются:

```text
deployment.apps/my-nginx-deployment-hpa created
service/my-loadbalancer-hpa created
horizontalpodautoscaler.autoscaling/my-hpa created
```

# 🧪 Практическая часть 8 — Тестирование нагрузки

Для наблюдения за компонентами кластера можно использовать команду.

Для macOS/Linux:

```bash
while true; do kubectl get pod,svc,hpa,nodes -o wide; sleep 5; done
```

Для Windows:

```bash
for /l %q in (0) do kubectl get pod,svc,hpa,nodes -o wide
```

---

## Создание нагрузки через wget

Для macOS/Linux:

```bash
while true; do wget -q -O- <IP-адрес_балансировщика>; done
```

Для Windows:

```bash
for /l %q in (0) do wget -q -O- <IP-адрес_балансировщика>
```

Ожидаемое поведение:

1. Сначала увеличится количество pod-ов.
2. Затем при нехватке ресурсов добавятся новые узлы.
3. После остановки нагрузки лишние pod-ы и узлы будут удалены.
