# 🐬 MySQL

## Что такое MySQL

**MySQL** — это реляционная система управления базами данных.

Она широко используется в веб-разработке, backend-сервисах, интернет-магазинах, CMS, внутренних корпоративных системах и других прикладных задачах.

---

## Преимущества MySQL

MySQL обладает несколькими важными преимуществами:

- гибкость и простота использования;
- открытый исходный код;
- поддержка Oracle и большого open-source-сообщества;
- высокая скорость обработки запросов;
- большое количество документации и примеров;
- широкая распространённость в индустрии;
- развитая система прав доступа;
- поддержка шифрования и механизмов защиты данных.

---

## Недостатки MySQL

Основные ограничения MySQL:

- возможные проблемы при работе с очень большими базами данных;
- неполная поддержка некоторых возможностей SQL;
- наличие собственных расширений;
- сложности развития и масштабирования в некоторых сценариях.

---

# 🛠️ Создание кластера MySQL

В практической части создаётся и настраивается кластер управляемой базы данных MySQL.

---

## Основные шаги

1. Перейти в консоль управления Yandex Cloud.
2. Выбрать каталог для кластера.
3. Нажать **Создать ресурс**.
4. Выбрать **Кластер MySQL**.
5. Задать имя кластера.
6. Выбрать окружение и версию.
7. Указать класс хостов.
8. Выбрать тип и размер хранилища.
9. Задать имя базы данных, пользователя и пароль.
10. Выбрать сеть.
11. Создать хост.
12. Включить публичный доступ.
13. Настроить дополнительные параметры.
14. Нажать **Создать кластер**.

---

## Типы хранилища

При создании кластера можно выбрать разные типы хранилища.

| Тип хранилища | Особенности |
|---|---|
| `network-hdd` | Самый экономичный вариант |
| `network-ssd` | Быстрее HDD, но дороже |
| `network-ssd-nonreplicated` | Повышенная производительность, но меньше избыточности |
| `local-ssd` | Самый быстрый вариант, но требует отказоустойчивого кластера |

Важно: размер хранилища можно изменить позже, а тип хранилища — нет.

## Создание таблицы ratings

```sql
CREATE TABLE IF NOT EXISTS ratings (
  rating_id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  message_id INT NOT NULL,
  rating INT NOT NULL
) ENGINE=INNODB;
```

---

## Добавление данных

```sql
INSERT INTO ratings (user_id, message_id, rating)
VALUES (44, 368, 4);
```

SQL-запросы через консоль управления удобны для небольших разовых задач, но для полноценной работы с БД лучше использовать клиентские приложения, backend-код или подключение через виртуальную машину.

# 🔌 Подключение к MySQL-кластеру

Есть два варианта подключения:

1. **Через интернет** — если у хоста открыт публичный доступ.  
   В этом случае используется защищённое SSL-соединение.

2. **Через виртуальную машину** — если публичный доступ закрыт.  
   ВМ должна находиться в той же виртуальной сети, что и кластер БД.

---

## Практический сценарий подключения

1. Создать новую ВМ на Ubuntu.
2. Подключиться к ней по SSH.
3. В Managed Service for MySQL открыть вкладку **Базы данных**.
4. Добавить новую БД, например `db2`.
5. Выдать пользователю права на доступ к этой БД.
6. Открыть инструкцию подключения.
7. Установить сертификат.
8. Установить утилиту `mysql`.
9. Подключиться к базе данных.
10. Создать таблицу.

---

## Пример таблицы users

```sql
CREATE TABLE IF NOT EXISTS users (
  user_id INT AUTO_INCREMENT,
  nickname VARCHAR(128) NOT NULL,
  avatar VARCHAR(255),
  mail VARCHAR(255),
```

# 💾 Резервное копирование и восстановление

Сервисы управляемых БД выполняют резервное копирование автоматически.

Для MySQL:

- резервное копирование выполняется один раз в сутки;
- журналы транзакций сохраняются;
- можно восстановить состояние кластера на момент в пределах периода хранения;
- по умолчанию резервные копии хранятся 7 дней;
- восстановление невозможно только для последних примерно 30 секунд.

---

## Восстановление из резервной копии

Основные шаги:

1. Открыть сервис **Managed Service for MySQL**.
2. Выбрать кластер.
3. Перейти на вкладку **Резервные копии**.
4. Нажать **Создать резервную копию**.
5. Напротив новой копии выбрать **Восстановить кластер**.
6. Задать параметры нового кластера.
7. Нажать **Восстановить кластер**.
8. Проверить, что таблицы и данные доступны.

# 🐘 PostgreSQL

## Что такое PostgreSQL

**PostgreSQL** — объектно-реляционная СУБД с открытым исходным кодом.

Она отличается высокой надёжностью, расширяемостью и часто используется:

- в финансовом учёте;
- в backend веб-сервисов;
- в корпоративных приложениях;
- в аналитических и интеграционных сценариях.

---

## Особенности PostgreSQL

В PostgreSQL таблицы могут рассматриваться как объекты, а операции с ними могут выполняться через объектно-ориентированные функции.

PostgreSQL поддерживает:

- сложные запросы;
- транзакции;
- расширения;
- строгую целостность данных;
- развитую систему индексов;
- работу с JSON;
- репликацию;
- гибкие типы данных.

---

## Преимущества PostgreSQL

- высокая надёжность;
- целостность данных;
- хорошая производительность на сложных запросах;
- расширяемость;
- открытый исходный код;
- большая поддержка сообщества;
- частые обновления;
- наличие Postgres Pro, соответствующего требованиям ФСТЭК.

---

## Недостатки PostgreSQL

- может уступать MySQL в простых операциях чтения;
- выше сложность настройки и администрирования;
- требуется более высокий уровень компетенции для эффективной работы.

# 📥 Импорт CSV-данных в PostgreSQL

## Подготовка ВМ

Для подключения к кластеру используется виртуальная машина на Ubuntu.

На ней устанавливается PostgreSQL-клиент:

```bash
sudo apt update && sudo apt install -y postgresql-client
```

---

## Создание CSV-файла

Пример файла `dtm.csv`:

```csv
"depot","store","time","distance"
"100","101",31,12
"100","102",38,17
"100","103",56,33
"100","104",70,60
"100","105",41,25
"100","106",21,8
"100","107",33,14
```

Файл содержит:

- `depot` — код склада;
- `store` — код пункта самовывоза;
- `time` — стандартное время доставки в минутах;
- `distance` — расстояние в километрах.

---

## Установка сертификата

```bash
mkdir -p ~/.postgresql
wget "https://storage.yandexcloud.net/cloud-certs/CA.pem" -O ~/.postgresql/root.crt
chmod 0600 ~/.postgresql/root.crt
```

---

## Подключение к PostgreSQL через SSL

```bash
psql "host=<FQDN_хоста> \
port=6432 \
sslmode=verify-full \
dbname=<имя_базы_данных> \
user=<имя_пользователя_базы_данных> \
target_session_attrs=read-write"
```

---

## Создание таблицы dtm

```sql
CREATE TABLE dtm (
  id serial PRIMARY KEY,
  depot int NOT NULL,
  store int NOT NULL,
  time int NOT NULL,
  distance int NOT NULL
);
```

---

## Загрузка данных из CSV

```sql
COPY dtm(depot, store, time, distance)
FROM '/<путь_к_файлу>/DTM.csv'
DELIMITERS ','
CSV HEADER;
```

Путь к файлу можно найти командой:

```bash
find / -name dtm.csv
```

После успешного импорта ожидаемый результат:

```text
COPY 7
```

**Greenplum** — аналитическая колоночная массивно-параллельная СУБД.

Greenplum используется в задачах:

- корпоративных хранилищ данных;
- ETL/ELT;
- сложной ad-hoc-аналитики;
- соединения больших таблиц;
- аналитических функций;
- Data Science;
- продуктовой аналитики;
- анализа логов.

---

## Особенности Greenplum

Greenplum быстро выполняет сложные аналитические запросы благодаря MPP-архитектуре.

Он сочетает:

- горизонтальную масштабируемость;

- отказоустойчивость;
- поддержку аналитических SQL-запросов;
- возможность работы с большими объёмами данных.

---

# 🧠 SMP и MPP

## SMP

**SMP** — symmetric multiprocessing.

Это архитектура, где несколько процессоров используют общую память и общие ресурсы.

---

## MPP

**MPP** — massive parallel processing.

Это архитектура, где данные и вычисления распределяются между независимыми узлами.

Greenplum использует подход `shared-nothing`, при котором каждый узел имеет собственные ресурсы.

# 🔁 Репликация

## Что такое репликация

**Репликация** — механизм синхронизации копий базы данных внутри кластера.

Она нужна для:

- отказоустойчивости;
- распределения нагрузки;
- резервирования данных;
- ускорения чтения;
- защиты от потери данных.

---

## Асинхронная репликация

При асинхронной репликации транзакция подтверждается на основном хосте до того, как данные полностью переданы на реплики.

Плюсы:

- выше скорость работы;
- меньше задержка при записи.

Минусы:

- при сбое часть данных может быть потеряна.

Такой подход реализован, например, в ClickHouse.

---

## Синхронная репликация

При синхронной репликации транзакция подтверждается только после записи данных на мастер и реплики.

Плюсы:

- выше надёжность;
- меньше риск потери данных.

Минусы:

- ниже скорость записи;
- возможное снижение производительности на 25–30%.

В PostgreSQL используется синхронная репликация.

---

## Физическая репликация

При физической репликации изменённые блоки данных побайтово копируются с диска одного хоста на диск другого.

К физической репликации относятся:

- синхронная репликация;
- асинхронная репликация.

---

## Логическая репликация

При логической репликации копируются не байтовые блоки, а таблицы и строки.

Используются:

- публикации;
- подписки.

Один экземпляр БД подписывается на изменения в другом экземпляре и получает от него данные.

Логическая репликация позволяет организовать:

- каскадную репликацию;
- снижение нагрузки на мастер;
- передачу данных между разными системами.

# 📌 Что важно запомнить

После лекции важно понимать:

- MySQL удобен для простых и распространённых backend-задач;
- PostgreSQL надёжен и хорошо подходит для сложных запросов и критичных систем;
- Greenplum используется для сложной аналитики по большим объёмам данных;
- managed-БД снижают нагрузку на администрирование;
- резервные копии позволяют восстановить состояние кластера;
- PostgreSQL поддерживает импорт данных через `COPY`;
- MPP-архитектура позволяет масштабировать аналитические вычисления;
- репликация бывает синхронной, асинхронной, физической и логической;
- Data Transfer используется для миграции, репликации и интеграции данных.
