# ☁️ Лекция 12 — Корпоративная аналитическая платформа: Data Transfer, ClickHouse, Object Storage и DataLens

Двенадцатая лекция посвящена построению **корпоративной аналитической платформы** в Yandex Cloud на примере интернет-магазина.

В рамках занятия рассматривается полный аналитический контур: от получения данных из операционной базы интернет-магазина до построения интерактивного дашборда в **Yandex DataLens**.  
Основной акцент сделан на синхронизации данных через **Yandex Data Transfer**, промежуточном хранении в **Managed Service for MySQL**, аналитической обработке в **Managed Service for ClickHouse**, использовании гибридного хранения с **Object Storage** и визуализации бизнес-метрик.

---

## 🎯 Цель лекции

Цель лекции — показать, как на практике построить облачную аналитическую инфраструктуру для бизнеса.

После изучения материала студент должен понимать:

- как устроен аналитический контур интернет-магазина;
- зачем нужны промежуточные хранилища данных;

- как использовать ClickHouse для аналитических запросов;
- зачем нужно гибридное хранение данных;
- как перемещать архивные данные в Object Storage;

# 🛒 Бизнес-сценарий: интернет-магазин

В лекции рассматривается пример интернет-магазина.

Покупатели размещают заказы через сайт или мобильное приложение, а отгрузка происходит из ближайшей выбранной точки.

Главная бизнес-задача:

> поддерживать товарные запасы на оптимальном уровне.

Это означает, что:

- товаров должно хватать для текущих заказов;
- покупатели должны получать товары вовремя;
- склады не должны быть переполнены;
- руководители должны видеть актуальную картину по заказам и остаткам.

---

## 📌 Задача аналитики

Необходимо создать дашборды с аналитическими отчётами, которые обновляются в реальном времени.

На основе этих отчётов руководители смогут принимать решения:

- где увеличить запасы;
- где уменьшить запасы;
- какие точки отгрузки перегружены;
- где наблюдается высокая концентрация заказов;
- как меняется динамика отгрузок по магазинам.

---

## ⚠️ Особенности исходных данных

Данные интернет-магазина находятся в разных местах:

- остатки товаров хранятся в корпоративном хранилище данных;
- заказы с сайта и мобильного приложения хранятся в MySQL или PostgreSQL;
- из-за технических ограничений в операционных системах хранится только часть истории;
- для аналитики нужен единый контур хранения и обработки данных.

---

# 🏗️ Общая архитектура решения

В лекции используется следующая логика построения платформы:

```text
Интернет-магазин → MySQL → Data Transfer → Managed MySQL → ClickHouse → DataLens
```

Дополнительно используется:

```text
ClickHouse → Object Storage
```

для хранения архивных данных.

---

## Основные компоненты

| Компонент | Назначение |
|---|---|
| Magento VM | Виртуальная машина с платформой интернет-магазина |
| MySQL на ВМ | Операционная база интернет-магазина |
| Yandex Data Transfer | Репликация данных из источника в облако |
| Managed Service for MySQL | Промежуточное облачное хранилище |
| Managed Service for ClickHouse | Аналитическая база данных |
| Object Storage | Хранение архивных данных |
| DataLens | Визуализация и построение дашбордов |

# 🔄 Модуль 1 — Синхронизация данных из MySQL

Первый этап — передать данные из MySQL интернет-магазина в промежуточное облачное хранилище.

---

## Шаг 1. Создание диска из образа Magento

Для начала создаётся виртуальный диск из заранее подготовленного образа с платформой интернет-магазина.

```bash
yc compute disk create \
  --zone ru-central1-a \
  --name web-store-lab-dataplatform \
  --source-image-id fd86cb7ugap89m9ja920 \
  --folder-id <your-yc-folder-id>
```

---

## Шаг 2. Создание виртуальной машины

После создания диска создаётся ВМ, на которой будет работать интернет-магазин.

```bash
yc compute instance create \
  --name magento \
  --zone ru-central1-a \
  --network-interface subnet-name=default-ru-central1-a,nat-ip-version=ipv4 \
  --hostname ya-sample-store \
  --use-boot-disk disk-name=web-store-lab-dataplatform \
  --folder-id <your-yc-folder-id> \
  --ssh-key ~/.ssh/id_rsa.pub
```

---

## Шаг 3. Подключение к ВМ

```bash
ssh yc-user@<ip-address-vm>
```

---

## Шаг 4. Просмотр схемы данных через DBeaver

Схему данных интернет-магазина можно посмотреть через DBeaver.

Параметры подключения:

```text
Server Host: <публичный_IP-адрес_виртуальной_машины>
Port: 3306
Database: ya_sample_store
User: magento-svc
Password: <password>
```

# 🔁 Настройка Yandex Data Transfer

Yandex Data Transfer используется для синхронизации данных между источником и приёмником.

В нашем случае:

```text
Источник: MySQL на ВМ интернет-магазина
Приёмник: Managed Service for MySQL
```

---

## Шаг 1. Создание endpoint источника

В сервисе **Data Transfer** создаётся endpoint источника.

Параметры:

```text
Имя: magento-source
Тип БД: MySQL
IP хоста: <публичный_IP-адрес_виртуальной_машины>
Имя базы данных: ya_sample_store
Имя пользователя: magento-svc
Пароль: <password>
Белый список таблиц: sales_*
```

Фильтр `sales_*` позволяет реплицировать только таблицы, связанные с заказами.

---

## Шаг 2. Создание endpoint приёмника

Далее создаётся endpoint приёмника — управляемая база данных MySQL в облаке.

Параметры:

```text
Имя: magento-report-dest
База данных: Managed Service for MySQL
Кластер: ya-sample-cloud-mysql
Имя базы данных: magento-cloud
Имя пользователя репликации: yc-user
Пароль: <password>
```

Дополнительно можно включить:

```text
Отключение проверки констрейнтов
```

Это нужно, чтобы избежать ошибок, если порядок передачи данных будет нарушен.

---

## Шаг 3. Создание трансфера

После создания endpoint-ов настраивается трансфер.

Параметры:

```text
Имя: sales-order-sync
Источник: magento-source
Приёмник: magento-report-dest
Тип трансфера: Копировать и реплицировать
```

После создания трансфер необходимо активировать.

---

## Что делает трансфер

После активации выполняется:

1. первоначальная синхронизация схемы данных;
2. копирование данных;
3. дальнейшая автоматическая репликация изменений.

Статус синхронизации и ошибки можно смотреть в разделе:

```text

```

## Проверка результата

# 📥 Модуль 2 — Импорт данных в ClickHouse

После синхронизации данных в Managed MySQL нужно перенести их в аналитическую базу ClickHouse.

---

## Логика импорта

Процесс состоит из четырёх шагов:

1. Создать кластер ClickHouse с гибридным хранилищем.
2. Создать интеграционную таблицу через `MySQL Engine`.
3. Создать аналитическую таблицу в ClickHouse.
4. Скопировать данные из внешней MySQL-таблицы в ClickHouse.

---

# 🟡 Создание ClickHouse-кластера

В сервисе **Managed Service for ClickHouse** создаётся кластер.

Основные параметры:

```text
Имя кластера: inventory
Имя базы данных: inventory
Имя пользователя: inventory
```

---

## Важные настройки ClickHouse

В дополнительных настройках необходимо включить:

- **Гибридное хранилище**
- **Доступ из DataLens**
- **Доступ из консоли управления**
- **Доступ из Метрики и AppMetrica**
- **Доступ из Serverless**

Гибридное хранилище позволяет хранить часто используемые данные на дисках ClickHouse, а архивные данные переносить в Object Storage.

# 🔗 Интеграционная таблица MySQL Engine

ClickHouse позволяет создавать таблицы, которые выглядят как обычные таблицы, но фактически обращаются к внешней MySQL-базе.

---

## Создание таблицы mysql_sales_order_grid

```sql
CREATE TABLE mysql_sales_order_grid (
  `entity_id` UInt32,
  `status` Nullable(String),
  `store_id` Nullable(UInt16),
  `store_name` Nullable(String),
  `customer_id` Nullable(UInt32),
  `grand_total` Nullable(Float32),
  `order_currency_code` String,
  `created_at` DateTime,
  `updated_at` DateTime,
  `billing_address` Nullable(String),
  `shipping_address` Nullable(String),
  `shipping_location` Nullable(String),
  `shipping_information` Nullable(String),
  `customer_email` Nullable(String),
  `customer_group` Nullable(String)
)
ENGINE = MySQL(
  '<my-yandex-cloud-cluster>.mdb.yandexcloud.net:3306',
  '<mysql-database-name>',
  'sales_order_grid',
  '<mysql-user>',
  '<mysql-user-pwd>'
);
```

## Проверка данных

# 📊 Аналитическая таблица sales_order

Далее создаётся полноценная таблица ClickHouse, куда будут загружены данные для аналитики.

---

## Создание таблицы sales_order

```sql
CREATE TABLE sales_order (
  entity_id UInt32,
  status Nullable(String),
  store_id Nullable(UInt16),
  store_name Nullable(String),
  customer_id Nullable(UInt32),
  grand_total Nullable(Float32),
  order_currency_code String,
  created_at DateTime,
  updated_at DateTime,
  billing_address Nullable(String),
  shipping_address Nullable(String),
  shipping_information Nullable(String),
  customer_email Nullable(String),
  customer_group Nullable(String),
  shipping_location Nullable(String)
)
ENGINE = ReplacingMergeTree()
PARTITION BY toYYYYMM(created_at)
ORDER BY (entity_id)
TTL created_at + INTERVAL 1 MONTH TO DISK 'object_storage';
```

---

## Что делает ReplacingMergeTree

`ReplacingMergeTree` позволяет исключать дубликаты записей.

Это важно при импорте и повторной синхронизации данных, когда одна и та же запись может попасть в таблицу несколько раз.

---

## Что делает TTL

`TTL` задаёт жизненный цикл данных.

В данной таблице правило означает:

```text
через 1 месяц после даты создания заказа запись переносится в Object Storage
```

Таким образом реализуется гибридное хранение:

- свежие данные — в ClickHouse;
- архивные данные — в Object Storage.

---

# 🔄 Копирование данных в ClickHouse

Для переноса данных из интеграционной MySQL-таблицы в таблицу ClickHouse используется запрос `INSERT INTO ... SELECT`.

```sql
INSERT INTO inventory.sales_order (
  entity_id,
  status,
  store_id,
  store_name,
  customer_id,
  grand_total,
  order_currency_code,
  created_at,
  updated_at,
  billing_address,
  shipping_address,
  shipping_location,
  shipping_information,
  customer_email,
  customer_group
)
SELECT
  entity_id,
  status,
  store_id,
  store_name,
  customer_id,
  grand_total,
  order_currency_code,
  created_at,
  updated_at,
  billing_address,
  shipping_address,
  shipping_location,
  shipping_information,
  customer_email,
  customer_group
FROM mysql_sales_order_grid
WHERE entity_id > (
  SELECT MAX(entity_id)
  FROM sales_order
);
```

```sql

```

В результате практики реализуется гибридное хранилище:


# 📈 Модуль 3 — Визуализация в DataLens

После подготовки данных в ClickHouse строится аналитический дашборд в DataLens.

---

## Создание подключения

На главной странице DataLens нужно:

1. нажать **Создать подключение**;
2. выбрать тип подключения **ClickHouse**;
3. указать параметры кластера;
4. указать имя хоста;
5. указать пользователя и пароль;
6. включить опцию **Разрешить использование подзапросов в качестве источника**;
7. нажать **Проверить подключение**;
8. сохранить подключение.

---

## Создание датасета

После подключения нужно:

1. нажать **Создать датасет**;
2. перенести на рабочую область таблицу:

```text
Inventory.mysql.sales_order_grid
```

3. перейти на вкладку **Поля**;
4. скорректировать типы и агрегации полей;
5. сохранить датасет.

---

```text
Геоточки (тепловая карта)
```

Для фильтрации по периоду нужно добавить поле:

```text
created_at
```

в секцию **Общие фильтры**.

Рекомендуемый пресет:

```text
Последние 365 дней
```

После настройки чарт нужно сохранить.

# 📝 Практическая часть лекции

В практической части студент выполняет следующие действия:

1. Создаёт диск из образа интернет-магазина.
2. Создаёт ВМ с Magento.
3. Подключается к ВМ по SSH.
4. Проверяет схему данных через DBeaver.
5. Создаёт кластер Managed Service for MySQL.
6. Создаёт endpoint источника в Data Transfer.
7. Создаёт endpoint приёмника в Data Transfer.
8. Создаёт трансфер типа **Копировать и реплицировать**.
9. Активирует трансфер.
10. Проверяет появление схемы данных в промежуточном MySQL.
11. Создаёт кластер Managed Service for ClickHouse.
