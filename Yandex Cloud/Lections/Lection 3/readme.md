# ☁️ Лекция 3 — Группы виртуальных машин, автомасштабирование и CLI в Yandex Cloud

Третья лекция посвящена работе с **группами виртуальных машин (Instance Groups)**, автоматическому восстановлению и масштабированию инфраструктуры, а также управлению ресурсами Yandex Cloud через **интерфейс командной строки CLI**.

В рамках занятия рассматриваются практические сценарии создания групп ВМ, настройки автомасштабирования, проведения стресс-тестов, работы с CLI-командами, использования файлов спецификаций YAML и базового подхода **Infrastructure as Code (IaC)**.

---

## 🎯 Цель лекции

Цель лекции — научиться автоматизировать управление облачной инфраструктурой и понимать, как создавать отказоустойчивые и масштабируемые сервисы на базе Yandex Cloud.

После изучения материала студент должен уметь:

- создавать группы виртуальных машин;
- настраивать фиксированный и автоматический размер группы;
- понимать принципы автоматического восстановления ВМ;
- тестировать автомасштабирование через нагрузку CPU;
- работать с Yandex Cloud CLI;
- создавать сети, подсети и ВМ через командную строку;
- использовать YAML-файлы спецификаций;
- создавать Instance Group через файл конфигурации;
- подключать группу ВМ к балансировщику нагрузки;
- обновлять и удалять группы ВМ.

---

## 🧩 Основные темы лекции

### 1. Группы виртуальных машин

**Instance Groups** — компонент сервиса Compute Cloud, который позволяет создавать группы виртуальных машин и централизованно управлять ими.

Группы ВМ позволяют:

- одновременно обновлять все машины в группе;
- равномерно распределять нагрузку между ВМ;
- интегрироваться с Network Load Balancer и Application Load Balancer;
- автоматически масштабировать количество машин;
- автоматически восстанавливать ВМ при сбоях;
- поддерживать надежную работу приложений в многозональной среде.

---

## 🔄 Автоматическое восстановление

Instance Groups регулярно проверяет состояние виртуальных машин и приложений.

Если ВМ становится неработоспособной, сервис может:

- создать новую ВМ вместо неисправной;
- перезагрузить существующую ВМ;
- восстановить целевой размер группы.

Поведение зависит от политики развёртывания:

- `max_expansion` — насколько можно превысить целевой размер группы;
- `max_unavailable` — сколько ВМ может быть временно недоступно.

---

## 📈 Автоматическое масштабирование

Автомасштабирование позволяет изменять количество ВМ в зависимости от нагрузки.

Основные параметры автомасштабирования:

| Параметр | Смысл |
|---|---|
| Минимальное количество ВМ | Нижняя граница размера группы |
| Максимальный размер группы | Верхняя граница размера группы |
| Целевая загрузка CPU | Уровень нагрузки, который сервис старается поддерживать |
| Период усреднения | Интервал, за который рассчитывается средняя нагрузка |
| Период прогрева | Время, в течение которого новая ВМ не учитывается в метриках |
| Период стабилизации | Задержка перед уменьшением количества ВМ после снижения нагрузки |

---

## 🧪 Стресс-тестирование автомасштабирования

Для проверки автомасштабирования используется утилита `stress`.

Установка:

```bash
sudo apt-get install stress
```

Запуск нагрузки на CPU:

```bash
stress -c 2
```

Аргумент `-c` означает нагрузку на процессор, а число после него задаёт количество ядер, которые будут нагружены.

После запуска стресс-теста Instance Groups отслеживает среднюю загрузку CPU и при превышении порога создаёт дополнительные виртуальные машины.

## 🖥️ Работа с Yandex Cloud CLI

**Yandex Cloud CLI** — это интерфейс командной строки для управления облачными ресурсами.

CLI подходит для:

- быстрого управления ресурсами;
- автоматизации рутинных задач;
- создания инфраструктуры через скрипты;
- работы с файлами спецификаций.

---

## ⚙️ Структура CLI-команды

Большинство команд `yc` строится по следующему принципу:

```bash
yc <service> <resource> <action> <parameters>
```

Пример:

```bash
yc resource-manager folder create --name project-1
```

Где:

- `resource-manager` — сервис;
- `folder` — ресурс;
- `create` — действие;
- `--name project-1` — параметр.

---

## 🔧 Настройка CLI

Основные шаги:

1. Установить Yandex Cloud CLI.
2. Запустить инициализацию:

```bash
yc init
```

3. Авторизоваться через браузер.
4. Выбрать облако, каталог и зону доступности.
5. Проверить список профилей:

```bash
yc config profile list
```

---

## 🔁 Перезагрузка ВМ через CLI

Получить справку по Compute Cloud:

```bash
yc compute --help
```

Получить справку по виртуальным машинам:

```bash
yc compute instance --help
```

Получить список ВМ:

```bash
yc compute instance list
```

Посмотреть справку по перезапуску:

```bash
yc compute instance restart --help
```

Перезапустить ВМ:

```bash
yc compute instance restart --name <имя_ВМ>
```

---

## 🌐 Создание сети и подсетей через CLI

Создание облачной сети:

```bash
yc vpc network create --name my-network
```

Создание подсети:

```bash
yc vpc subnet create \
  --name my-subnet-1 \
  --zone ru-central1-a \
  --range 192.168.1.0/24 \
  --network-name my-network
```

Аналогично создаются подсети в других зонах доступности:

- `ru-central1-a`
- `ru-central1-b`
- `ru-central1-c`

## 🖥️ Создание ВМ через CLI

Пример создания виртуальной машины:

```bash
yc compute instance create \
  --name my-instance-1 \
  --hostname my-instance-1 \
  --zone ru-central1-a \
  --create-boot-disk image-family=ubuntu-2004-lts,size=30,type=network-nvme \
  --image-folder-id standard-images \
  --memory 4 \
  --cores 2 \
  --core-fraction 100 \
  --network-interface subnet-name=my-subnet-1,nat-ip-version=ipv4 \
  --async
```

Проверка списка созданных ВМ:

```bash
yc compute instance list
```

Вывод списка в формате JSON:

```bash
yc compute instance list --format json
```

Удаление ВМ:

```bash
yc compute instance delete my-instance-1 my-instance-2 my-instance-3
```

## 🧾 Infrastructure as Code

**Infrastructure as Code (IaC)** — это подход, при котором инфраструктура описывается не вручную через интерфейс, а с помощью конфигурационных файлов.

Основная идея:

> инфраструктура описывается как код и может храниться, изменяться и контролироваться так же, как программный код.

Преимущества IaC:

- повторяемость конфигураций;
- автоматизация развёртывания;
- контроль версий;
- снижение числа ручных ошибок;
- удобство командной работы;
- быстрое восстановление инфраструктуры.

---

## 📄 YAML-файлы спецификаций

Для описания инфраструктуры используется YAML.

**YAML** — это формат сериализации данных с читаемым синтаксисом.

Преимущества YAML:

- удобен для чтения человеком;
- независим от языка программирования;
- подходит для хранения сложных конфигураций;
- хорошо работает с системами контроля версий;
- требует строгого соблюдения отступов.

---

## 🧱 Создание Instance Group через спецификацию

Для создания группы ВМ используется файл, например:

```text
specification.yaml
```

В спецификации указываются:

- имя группы;
- сервисный аккаунт;
- шаблон виртуальной машины;
- параметры диска;
- сетевые настройки;
- политика развёртывания;
- политика масштабирования;
- распределение по зонам;
- настройки балансировщика.

---

## 🧩 Пример структуры спецификации

```yaml
name: my-group
service_account_id: <идентификатор_сервисного_аккаунта>

instance_template:
  platform_id: standard-v3
  resources_spec:
    memory: 2g
    cores: 2

  boot_disk_spec:
    mode: READ_WRITE
    disk_spec:
      image_id: <идентификатор_образа>
      type_id: network-hdd
      size: 32g

  network_interface_specs:
    - network_id: <идентификатор_сети>
      primary_v4_address_spec:
        one_to_one_nat_spec:
          ip_version: IPV4

deploy_policy:
  max_unavailable: 1
  max_expansion: 0

scale_policy:
  fixed_scale:
    size: 3

allocation_policy:
  zones:
    - zone_id: ru-central1-a
```

## ⚙️ Cloud-init

Для автоматической установки ПО при первом запуске ВМ используется `cloud-init`.

Пример установки NGINX и изменения стартовой страницы:

```yaml
metadata:
  user-data: |-
    #cloud-config
    package_update: true
    runcmd:
      - [apt-get, install, -y, nginx]
      - [/bin/bash, -c, 'source /etc/lsb-release; sed -i "s/Welcome to nginx/It is $(hostname) on $DISTRIB_DESCRIPTION/" /var/www/html/index.nginx-debian.html']
```

Такой подход позволяет не подключаться к каждой машине вручную и сразу получать готовые веб-серверы.

## 🚀 Создание группы ВМ по спецификации

Команда создания Instance Group:

```bash
yc compute instance-group create --file <путь_к_файлу_specification.yaml>
```

Проверка списка групп:

```bash
yc compute instance-group list
```

## ⚖️ Создание Network Load Balancer через CLI

Создание балансировщика:

```bash
yc load-balancer network-load-balancer create \
  --region-id ru-central1 \
  --name my-load-balancer \
  --listener name=my-listener,external-ip-version=ipv4,port=80
```

Получение списка целевых групп:

```bash
yc load-balancer target-group list
```

Подключение целевой группы к балансировщику:

```bash
yc load-balancer network-load-balancer attach-target-group my-load-balancer \
  --target-group target-group-id=<идентификатор_целевой_группы>,healthcheck-name=test-health-check,healthcheck-interval=2s,healthcheck-timeout=1s,healthcheck-unhealthythreshold=2,healthcheck-healthythreshold=2,healthcheck-http-port=80
```

---

## 🩺 Проверка состояния машин группы

Проверить состояние машин, подключённых к балансировщику:

```bash
yc load-balancer network-load-balancer target-states my-load-balancer \
  --target-group-id <идентификатор_целевой_группы>
```

Нужно дождаться статуса:

```text
HEALTHY
```

После этого можно открыть внешний IP-адрес балансировщика в браузере и убедиться, что страница доступна.

---

## 🔄 Обновление Instance Group

В лекции рассматривается обновление группы ВМ через изменение файла спецификации.

Например, можно заменить образ Ubuntu 18.04 LTS на Ubuntu 20.04 LTS.

Команда обновления:

```bash
yc compute instance-group update \
  --name my-group \
  --file <путь_к_файлу_specification.yaml>
```

После обновления нужно проверить, что ВМ пересозданы и на приветственной странице отображается новая версия ОС.

---

## 🧪 Проверка отказоустойчивости

Для проверки отказоустойчивости можно удалить одну ВМ из группы:

```bash
yc compute instance delete <имя_ВМ>
```

Ожидаемое поведение:

- балансировщик переключит трафик на другую машину;
- страница останется доступной;
- Instance Groups автоматически пересоздаст удалённую ВМ.

---

## 🧹 Удаление ресурсов

После завершения практики необходимо удалить созданные ресурсы, чтобы избежать лишних расходов.

Удаление группы ВМ:

```bash
yc compute instance-group delete --name my-group
```

Удаление балансировщика:

```bash
yc load-balancer network-load-balancer delete --name my-load-balancer
```

## 📌 Что важно запомнить

После лекции важно понимать:

- группы ВМ позволяют централизованно управлять набором машин;
- Instance Groups могут автоматически восстанавливать неисправные ВМ;
- автомасштабирование помогает адаптировать инфраструктуру к нагрузке;
- CLI позволяет управлять облаком быстрее и удобнее, чем через веб-интерфейс;
- YAML-спецификации позволяют описывать инфраструктуру как код;
- cloud-init позволяет автоматически устанавливать ПО при первом запуске ВМ;
- балансировщик нагрузки повышает доступность сервиса;
- удаление одной ВМ не должно останавливать сервис, если инфраструктура настроена правильно.
