# ☁️ Лекция 4 — Packer, Terraform, DevOps и инструменты автоматизации в Yandex Cloud

Четвёртая лекция посвящена практическим инструментам автоматизации облачной инфраструктуры и современным DevOps-подходам.

В рамках занятия рассматриваются **Packer**, **Terraform**, принципы **Infrastructure as Code**, основы **DevOps**, работа с системами контроля версий, CI/CD, GitLab, Container Registry, мониторинг, логирование и инструменты для непрерывной доставки приложений.

---

## 🎯 Цель лекции

Цель лекции — показать, как перейти от ручного создания облачных ресурсов к автоматизированному управлению инфраструктурой и приложениями.

После изучения материала студент должен понимать:

- что такое SCM, Git, CI/CD;

- зачем нужен Container Registry;
- как используются мониторинг, логирование и нагрузочное тестирование.

---

## 🧩 Основные темы лекции

Лекция охватывает следующие блоки:

1. **Packer** — автоматизация сборки образов ВМ.

# 🧱 Packer

## Что такое Packer

**Packer** — это инструмент, который стандартизирует и автоматизирует процесс создания образов виртуальных машин.

Он позволяет заранее подготовить образ с нужной операционной системой, установленным ПО и базовой конфигурацией.

---

## Зачем нужен Packer

Packer используется для того, чтобы:

- сократить время создания новых инстансов;
- обеспечить одинаковую конфигурацию окружений;
- автоматизировать сборку образов;
- уменьшить количество ручных действий;
- быстрее запускать виртуальные машины с уже установленным ПО.

---

## Пример задачи с Packer

В лекции рассматривается создание пользовательского образа Ubuntu с установленным NGINX.

Пример файла спецификации:

```hcl
source "yandex" "ubuntu-nginx" {
  token               = "<OAuth-токен>"
  folder_id           = "<идентификатор_каталога>"
  source_image_family = "ubuntu-2004-lts"
  ssh_username        = "ubuntu"
  use_ipv4_nat        = "true"
  image_description   = "my custom ubuntu with nginx"
  image_family        = "ubuntu-2004-lts"
  image_name          = "my-ubuntu-nginx"
  subnet_id           = "<идентификатор_подсети>"
  disk_type           = "network-ssd"
  zone                = "ru-central1-a"
}

build {
  sources = ["source.yandex.ubuntu-nginx"]

  provisioner "shell" {
    inline = [
      "sudo apt-get update -y",
      "sudo apt-get install -y nginx",
      "sudo systemctl enable nginx.service"
    ]
  }
}
```

---

## Сборка образа

Команда для создания образа:

```bash
packer build <путь_к_файлу_my-ubuntu-nginx.pkr.hcl>
```

После выполнения команды образ появляется в разделе **Compute Cloud → Образы**.

---

## Практический результат работы с Packer

После выполнения практики студент должен уметь:

- установить Packer;
- подготовить HCL-файл спецификации;
- создать пользовательский образ ВМ;
- установить ПО в образ автоматически;
- создать ВМ из пользовательского образа;
- проверить работу веб-сервера по публичному IP-адресу.

# 🏗️ Terraform

## Что такое Terraform

**Terraform** — это инструмент с открытым исходным кодом для управления облачной инфраструктурой в декларативном стиле.

Terraform реализует подход **Infrastructure as Code (IaC)**, при котором инфраструктура описывается не вручную, а в виде конфигурационных файлов.

---

## Зачем нужен Terraform

Terraform используется для:

- быстрого развёртывания инфраструктуры;
- описания ресурсов в виде кода;
- проверки плана изменений перед применением;
- снижения количества ошибок при ручной настройке;
- управления инфраструктурой в разных облачных провайдерах;
- воспроизводимости окружений.

---

## Три этапа работы с Terraform

Основной цикл работы с Terraform состоит из трёх команд.

### 1. Инициализация

```bash
terraform init
```

Команда инициализирует провайдеров, указанных в файле спецификации.

---

### 2. Планирование

```bash
terraform plan
```

Команда показывает, какие ресурсы будут созданы, изменены или удалены.

---

### 3. Применение

```bash
terraform apply
```

Команда запускает создание или изменение инфраструктуры.

---

## State-файл

Terraform хранит состояние инфраструктуры в специальном state-файле.

State-файл — это реестр в формате JSON, в котором фиксируется текущее состояние ресурсов в облаке.

Посмотреть список ресурсов в state можно командой:

```bash
terraform state list
```

---

## Пример настройки провайдера Yandex Cloud

```hcl
terraform {
  required_providers {
    yandex = {
      source = "yandex-cloud/yandex"
    }
  }
}

provider "yandex" {
  token     = "<OAuth-токен>"
  cloud_id  = "<идентификатор_облака>"
  folder_id = "<идентификатор_каталога>"
  zone      = "<зона_доступности_по_умолчанию>"
}
```

---

## Пример создания виртуальной машины

```hcl
resource "yandex_compute_instance" "vm-1" {
  name        = "from-terraform-vm"
  platform_id = "standard-v1"
  zone        = "ru-central1-a"

  resources {
    cores  = 2
    memory = 2
  }

  boot_disk {
    initialize_params {
      image_id = "fd8fosbegvnhj5haiuoq"
    }
  }

  network_interface {
    subnet_id = yandex_vpc_subnet.subnet-1.id
    nat       = true
  }

  metadata = {
    ssh-keys = "ubuntu:${file("~/.ssh/id_rsa.pub")}"
  }
}
```

---

## Пример создания сети и подсети

```hcl
resource "yandex_vpc_network" "network-1" {
  name = "from-terraform-network"
}

resource "yandex_vpc_subnet" "subnet-1" {
  name           = "from-terraform-subnet"
  zone           = "ru-central1-a"
  network_id     = yandex_vpc_network.network-1.id
  v4_cidr_blocks = ["10.2.0.0/16"]
}
```

---

## Outputs

Terraform может выводить значения созданных ресурсов, например внутренний и внешний IP-адрес ВМ:

```hcl
output "internal_ip_address_vm_1" {
  value = yandex_compute_instance.vm-1.network_interface.0.ip_address
}

output "external_ip_address_vm_1" {
  value = yandex_compute_instance.vm-1.network_interface.0.nat_ip_address
}
```

---

## Удаление инфраструктуры

Для удаления созданной инфраструктуры используется команда:

```bash
terraform destroy
```

# 🔁 DevOps

## Классическая проблема Dev и Ops

В лекции рассматривается конфликт между двумя ролями:

### Development

Команда разработки:

- создаёт новые функции;
- отвечает за изменения;
- ориентируется на закрытые задачи и внедрённые фичи;
- часто не отвечает за стабильность после релиза.

### Operations

Команда эксплуатации:

- отвечает за стабильность;
- поддерживает работоспособность сервисов;
- ориентируется на SLA и безаварийную работу;
- работает с продуктивной средой.

---

## Идея DevOps

DevOps предлагает объединить разработку и эксплуатацию, чтобы перейти:

- от конфронтации к сотрудничеству;
- от ручных релизов к автоматизации;
- от долгих циклов внедрения к быстрым поставкам;
- от разрозненных команд к единой ответственности за сервис.

---

## Принцип You Build It, You Run It

Один из ключевых принципов DevOps:

> Кто разработал сервис, тот участвует в его поддержке.

Преимущества:

- меньше перекладывания ответственности;
- выше качество сервиса;
- больше мотивации автоматизировать рутину;
- разработчики лучше понимают инфраструктуру.

Недостатки:

- разработчики могут тратить время на эксплуатационные задачи;
- нужны дополнительные компетенции;
- необходима организация дежурств и работы с инцидентами.

---

## Критика DevOps

В лекции также обсуждаются ограничения DevOps:

- иногда DevOps остаётся декларацией без практической реализации;
- безопасность часто требует отдельного подхода DevSecOps;
- DevOps не всегда покрывает низкоуровневую эксплуатацию;
- часто путают DevOps и CI/CD.

# 🧬 SCM и Git

## SCM / VCS

**SCM (Source Code Management)** или **VCS (Version Control System)** — система управления исходным кодом.

SCM позволяет:

- хранить историю изменений;
- фиксировать изменения через коммиты;
- работать с ветками;
- использовать pull request;
- проводить code review;
- синхронизировать изменения через удалённый репозиторий.

---

## Git

**Git** — распределённая система контроля версий.

Особенности Git:

- у каждого разработчика есть локальная копия всего репозитория;
- можно работать с несколькими удалёнными репозиториями;
- поддерживаются разные модели работы:
  - git-flow;
  - github-flow;
  - trunk-based development;
- Git хорошо интегрируется со средствами разработки и DevOps-платформами.

---

## Удалённый репозиторий

Удалённый репозиторий используется для:

- совместной работы команды;
- хранения центральной версии проекта;
- разделения прав доступа;
- визуализации изменений;
- интеграции с трекером задач;
- запуска CI/CD-пайплайнов.

---

# ⚙️ CI/CD

## Continuous Integration

**CI (Continuous Integration)** — это автоматизация сборки и тестирования проекта.

Обычно процесс выглядит так:

1. Разработчик отправляет код в удалённый репозиторий.
2. CI-система получает изменения.
3. CI-система скачивает исходный код.
4. Выполняется сборка проекта.
5. Запускаются тесты.
6. Формируется отчёт.
7. Готовая сборка загружается в реестр.

---

## Архитектура CI

Типовая архитектура CI включает:

- управляющий сервис CI;
- воркеры;
- репозиторий исходного кода;
- registry для хранения артефактов или сборок.

CI-сервис получает задания по команде пользователя, по расписанию или при изменениях в репозитории.

---

## Continuous Delivery и Continuous Deployment

### Continuous Delivery

Автоматизация доставки приложения по команде ответственного исполнителя.

### Continuous Deployment

Полностью автоматический процесс доставки в продуктивную среду после принятия изменений в основную ветку.

---

## Связка CI/CD/DevOps

В лекции показана цепочка:

```text
Plan → Code → Build → Test → Release → Deploy → Operate → Monitor
```

Эта цепочка объединяет:

- Continuous Integration;
- Continuous Delivery;
- Continuous Deployment;
- DevOps.

---

## Примеры CI/CD-инструментов

В лекции упоминаются:

- CruiseControl;
- Jenkins;
- GitLab CI;
- Travis CI;
- TeamCity;
- Bamboo;
- CircleCI;
- Azure DevOps.

# 🦊 GitLab

## Что такое GitLab

**GitLab** — это веб-инструмент жизненного цикла DevOps с открытым исходным кодом.

В основе GitLab лежит Git SCM и дополнительные сервисы:

- Wiki;
- Issue Tracker;
- CI/CD;
- управление репозиториями;
- управление правами;
- реестры пакетов;
- интеграции с Kubernetes, Terraform, Monitoring и другими инструментами.

---

## GitLab SCM

GitLab может использоваться как:

- self-hosted-решение;
- облачный сервис gitlab.com;
- управляемый сервис Managed GitLab в Yandex Cloud.

Преимущества:

- бесплатная Community Edition;
- встроенный WebIDE;
- встроенный CI;
- встроенная Wiki;
- встроенный Issue Tracker;
- встроенные реестры пакетов;
- гибкая система прав доступа;
- поддержка gitlab-runner.

---

## GitLab CI/CD

В GitLab CI/CD сценарий описывает:

- этапы сборки;
- задания;
- зависимости между задачами;
- правила запуска пайплайна.

Обычно CI/CD-конфигурация хранится в файле:

```text
.gitlab-ci.yml
```

---

## GitLab Runner

**GitLab Runner** — компонент, который выполняет CI/CD-задания.

Runner может работать с разными executor-ами:

- shell;
- ssh;
- docker;
- virtualbox;
- kubernetes.

Runner-ы можно распределять по проектам и типам задач с помощью тегов.

---

## Managed GitLab

Managed GitLab — полностью управляемый сервис GitLab.

Он позволяет:

- развернуть GitLab в один клик;
- не тратить время на поддержку и обновления;
- использовать Community Edition;
- интегрироваться с Managed Kubernetes;
- интегрироваться с Yandex Tracker;
- при необходимости перейти на Enterprise Edition.

---

# 📦 Container Registry

**Container Registry** — управляемый сервис Yandex Cloud для хранения и распространения Docker-образов.

Основные возможности:

- хранение Docker-образов;
- адреса вида `cr.yandex/...`;
- интеграция с IAM;
- политики очистки;
- инструменты безопасности;
- использование Object Storage в качестве основы хранения.

---

# 🧪 Load Testing

**Load Testing** — инструмент нагрузочного тестирования, разработанный в Yandex.

Он используется для проверки:

- производительности приложения;
- устойчивости сервиса под нагрузкой;
- поведения инфраструктуры при росте числа запросов;
- возможностей масштабирования.

---

# 🚀 ArgoCD

**ArgoCD** — инструмент непрерывной доставки и деплоя с открытым исходным кодом.

Особенности ArgoCD:

- декларативная конфигурация;
- удобный веб-интерфейс;
- возможность управлять несколькими Kubernetes-кластерами;
- ручная, pull- и callback-активация;
- уведомления об изменении состояния;
- ориентация на Kubernetes.

Важно: ArgoCD не работает без Kubernetes.

# 🛠️ Terraform и Crossplane

## Terraform

Terraform рассматривается как популярный инструмент **Infrastructure as Code**.

Он позволяет описывать инфраструктуру декларативно и управлять облачными ресурсами через конфигурационные файлы.

## Crossplane

Crossplane рассматривается как перспективный инструмент IaC, особенно в Kubernetes-ориентированной инфраструктуре.

# 📊 Monitoring, Logging, OpenSearch

## Yandex Monitoring

**Yandex Monitoring** — система сбора, хранения и отображения метрик.

Она позволяет:

- собирать метрики облачных ресурсов;
- строить графики;
- настраивать алерты;
- отправлять уведомления;
- добавлять собственные метрики через Unified Agent.

По умолчанию для ВМ доступна информация через гипервизор:

- загрузка CPU;
- дисковые операции;
- сетевые метрики.

Для более глубокой диагностики можно установить **Unified Agent**.

---

## Cloud Logging

**Cloud Logging** — сервис централизованного сбора логов в Yandex Cloud.

Он нужен для:

- хранения логов приложений;
- анализа событий;
- поиска ошибок;
- диагностики инцидентов.

---

## OpenSearch

**OpenSearch** — управляемый сервис для сбора, хранения и анализа логов и других данных.

Он может использоваться для:

- полнотекстового поиска;
- анализа логов;
- построения observability-платформ;
- мониторинга событий в инфраструктуре.

# 📝 Практическая часть лекции

В практической части лекции студент выполняет следующие действия:

1. Устанавливает Packer.
2. Создаёт HCL-спецификацию образа.
3. Собирает пользовательский образ Ubuntu с NGINX.
4. Создаёт ВМ из пользовательского образа.
5. Устанавливает Terraform.
6. Настраивает провайдер Yandex Cloud.
7. Описывает ВМ, сеть и подсеть в Terraform.
8. Выполняет `terraform init`.
9. Выполняет `terraform plan`.
10. Выполняет `terraform apply`.
11. Проверяет state-файл.
12. Изменяет спецификацию.
13. Удаляет инфраструктуру через `terraform destroy`.
14. Изучает DevOps-подход и инструменты CI/CD.
15. Разбирает GitLab, GitLab Runner, Container Registry, Monitoring и Logging.

# 📌 Что важно запомнить

После лекции важно понимать:

- Packer нужен для стандартизированной сборки образов ВМ;
- Terraform нужен для декларативного управления инфраструктурой;
- Infrastructure as Code снижает количество ручных ошибок;
- DevOps объединяет разработку и эксплуатацию;
- CI/CD автоматизирует сборку, тестирование и доставку приложений;
- GitLab может выступать полноценной DevOps-платформой;
- Container Registry хранит Docker-образы;
- мониторинг и логирование являются обязательной частью эксплуатации;
- ArgoCD используется для непрерывной доставки в Kubernetes.

Главная идея лекции:

> инфраструктура должна описываться кодом, приложения должны доставляться автоматически, а работа сервисов должна постоянно наблюдаться через мониторинг и логи.

Эта лекция связывает облачные технологии, DevOps, IaC, CI/CD и практические инструменты Yandex Cloud в единую систему.
