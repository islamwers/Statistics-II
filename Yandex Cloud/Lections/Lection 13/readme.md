# ⚡ Serverless

## Что такое Serverless

**Serverless** — это бессерверный подход к разработке приложений, при котором разработчик пишет код и настраивает бизнес-логику, но не управляет серверами, виртуальными машинами, контейнерами и инфраструктурой вручную.

В Yandex Cloud serverless-сервисы позволяют:

- запускать программный код;
- хранить данные;
- создавать интеграции между сервисами;
- автоматически реагировать на события;
- масштабировать обработку запросов без ручной настройки инфраструктуры.

---

## Преимущества Serverless

Использование serverless-сервисов даёт несколько важных преимуществ.

### 1. Фокус на коде

Разработчику не нужно разворачивать и обслуживать инфраструктуру.

Не требуется вручную управлять:

- серверами;
- виртуальными машинами;
- контейнерами;
- системными обновлениями;
- масштабированием вычислительных ресурсов.

---

### 2. Снижение операционных расходов

Serverless может быть дешевле классического запуска приложений на виртуальных машинах, потому что ресурсы используются только тогда, когда выполняется функция.

Это особенно удобно для задач:

- с нерегулярной нагрузкой;
- с короткими вычислениями;
- с событийной обработкой;
- с автоматизацией внутренних процессов.

---

### 3. Гибкость при непредсказуемой нагрузке

Serverless помогает снизить риски неправильной оценки будущей нагрузки.

Если спрос неожиданно вырастет, облачная платформа сможет автоматически обработать больше вызовов.

Если спрос окажется ниже ожидаемого, не придётся переплачивать за простаивающие серверы.

# 🧠 Yandex Cloud Functions

## Что такое Yandex Cloud Functions

**Yandex Cloud Functions** — это сервис категории **Function as a Service (FaaS)**.

Его основная задача — запускать программный код, написанный на одном из поддерживаемых языков программирования.

Функции выполняются в специально подготовленной среде выполнения — runtime.

---

## Что умеет Cloud Functions

Yandex Cloud Functions позволяет:

- запускать код по HTTP-вызову;
- запускать код по событиям из других сервисов;
- вести журнал выполнения функций;
- создавать версии функций;
- работать с переменными окружения;
- подключать SDK и зависимости;
- интегрироваться с другими сервисами Yandex Cloud.

---

## Журналы выполнения

Cloud Functions ведёт журнал выполнения функций.

В логах можно увидеть:

- параметры запуска функции;
- время запуска;
- результат выполнения;
- ошибки;
- пользовательские сообщения из кода;
- данные для отладки.

Логи помогают понять, почему функция не сработала или почему данные не были записаны в нужный сервис.

---

## Интеграция с другими сервисами

Cloud Functions может запускаться не только вручную, но и автоматически.

Для этого используются **триггеры**.

Например, функцию можно запустить:

- при появлении сообщения в Message Queue;
- при загрузке объекта в Object Storage;
- при получении данных от IoT-устройства;
- при создании Docker-образа в Container Registry.

---

## SDK для разработчиков

В функциях можно использовать SDK и библиотеки для взаимодействия с сервисами Yandex Cloud.

Например, функция может:

- записывать файл в Object Storage;
- читать данные из базы;
- отправлять сообщение в очередь;
- обрабатывать входящий HTTP-запрос;
- запускать другую часть облачного сценария.

---

# ⚠️ Ограничения Cloud Functions

У Cloud Functions есть важные ограничения, которые нужно учитывать при проектировании приложений.

---

## 1. Функции должны быть конечными

Функция не может выполняться бесконечно.

У каждой функции есть ограничение по времени выполнения.

Если функция работает слишком долго, она будет автоматически завершена.

---

## 2. Функции не хранят состояние

Cloud Functions — это stateless-среда.

Это означает:

- функция не должна рассчитывать на сохранение локального состояния;
- разные вызовы функции могут выполняться в разных экземплярах;
- экземпляры функции не обмениваются локальной памятью;
- переменные в памяти нельзя использовать как постоянное хранилище.

Если нужно сохранить состояние, данные следует записывать во внешнее хранилище:

- Object Storage;
- YDB;
- Managed Database;
- Message Queue;
- другое постоянное хранилище.

# 🛠️ Создание функции через веб-интерфейс

## Шаги создания функции

1. Открыть консоль управления Yandex Cloud.
2. В списке сервисов выбрать **Cloud Functions**.
3. Нажать **Создать функцию**.
4. Указать имя функции.
5. Добавить краткое описание.
6. Выбрать среду выполнения кода.
7. Выбрать способ загрузки кода.
8. Указать точку входа.
9. Нажать **Создать версию**.

---

## Способы загрузки кода

Код функции можно добавить несколькими способами:

- через редактор кода в веб-интерфейсе;
- загрузкой ZIP-архива с локальной машины;
- загрузкой файла из Object Storage.

Если размер кода больше ограничения веб-редактора, удобнее использовать ZIP-архив или Object Storage.

---

## Точка входа

Точка входа определяет, какая функция будет вызвана первой.

Формат точки входа:

```text
<имя_файла>.<имя_обработчика>
```

Пример:

```text
index.handler
```

или

```text
index.Handler
```

# 🧪 Тестирование функции

После создания версии функции её можно протестировать во вкладке **Тестирование**.

Для проверки HTTP-вызова нужно:

1. открыть вкладку **Тестирование**;
2. выбрать шаблон данных **HTTPS-вызов**;
3. убедиться, что входные данные сформированы в формате JSON;
4. нажать **Запустить тест**;
5. посмотреть ответ функции.

Функция возвращает результат также в формате JSON.

# 💻 Запуск функции через CLI

Cloud Functions можно создавать и запускать через командную строку `yc`.

---

## Шаг 1. Создать сервисный аккаунт

Создаётся сервисный аккаунт для работы с функциями.

```bash
yc iam service-account create --name service-account-for-cf
```

---

## Шаг 2. Назначить роль сервисному аккаунту

Для учебной практики сервисному аккаунту назначается роль `editor`.

```bash
yc resource-manager folder add-access-binding <folder_id> \
  --role editor \
  --subject serviceAccount:<service_account_id>
```

---

## Шаг 3. Создать функцию

```bash
yc serverless function create --name my-first-function
```

После создания функция будет непубличной.

---

## Шаг 4. Создать файл с кодом функции

```bash
nano index.py
```

Пример кода:

```python
def handler(event, context):
    return {
        'statusCode': 200,
        'body': 'Hello World!',
    }
```

---

## Шаг 5. Создать версию функции

```bash
yc serverless function version create \
  --function-name my-first-function \
  --memory 256m \
  --execution-timeout 5s \
  --runtime python37 \
  --entrypoint index.handler \
  --service-account-id $SERVICE_ACCOUNT_ID \
  --source-path index.py
```

---

## Шаг 6. Проверить список функций

```bash
yc serverless function list
```

---

## Шаг 7. Проверить версии функции

```bash
yc serverless function version list --function-name my-first-function
```

---

## Шаг 8. Вызвать функцию через CLI

```bash
yc serverless function invoke <function_id>
```

# 📦 Как устроена функция

Функция в Cloud Functions — это логический контейнер с параметрами.

Внутри функции запускается конкретная версия пользовательского кода.

---

## Основные элементы функции

Функция содержит:

- имя;
- описание;
- уникальный идентификатор;
- среду выполнения;
- точку входа;
- таймаут;
- объём памяти;
- переменные окружения;
- версии кода;
- теги версий.

---

## Обработчик

Обработчик — это метод, который вызывается при запуске функции.

Он принимает:

- входные данные;
- контекст вызова.

Пример:

```python
def handler(event, context):
    ...
```

---

## Контекст вызова

Контекст используется для взаимодействия кода функции с Cloud Functions.

Например, через контекст можно получить служебную информацию о вызове функции и времени выполнения.

---

# 🧬 Версии функций и теги

## Версии

Каждое сохранённое состояние кода можно зафиксировать как новую версию функции.

Для каждой версии сохраняется уникальный идентификатор.

Версии позволяют:

- хранить историю изменений;
- быстро откатываться к старому коду;
- тестировать новые варианты функции;
- управлять стабильными и экспериментальными версиями.

---

## Тег `$latest`

Последняя созданная версия автоматически получает тег:

```text
$latest
```

Когда создаётся новая версия, тег `$latest` переносится на неё.

---

## Пользовательские теги

Можно назначить собственный тег конкретной версии.

Пример:

```bash
yc serverless function version set-tag \
  --id <version_id> \
  --tag first
```

Теги позволяют вызывать определённую версию функции и удобно управлять рабочими сценариями.

# 🔔 Триггеры

## Что такое триггер

**Триггер** — это условие, при выполнении которого автоматически запускается функция.

Общая логика:

```text
Событие в сервисе → Триггер → Cloud Function
```

---

## Какие сервисы могут запускать функции

Триггеры позволяют запускать функции по событиям из разных сервисов:

- **Yandex Message Queue** — при появлении нового сообщения;
- **IoT Core** — при получении данных от устройства;
- **Object Storage** — при создании, изменении ACL или удалении объекта;
- **Container Registry** — при создании или удалении Docker-образа или тега;
- **Cloud Logs** — при появлении логов.

---

# 🪣 Триггер от Object Storage

В практической части создаётся сценарий, где функция автоматически запускается при появлении нового объекта в бакете.

Итоговая цепочка выглядит так:

```text
Вызов my-first-function
        ↓
Создание файла в Object Storage
        ↓
Срабатывание my-first-trigger
        ↓
Запуск my-trigger-function
        ↓
Запись события event в логи
```

---

# 🔐 Подготовка доступа к Object Storage

## Шаг 1. Назначить роль сервисному аккаунту

Для работы с Object Storage сервисному аккаунту нужна роль:

```text
storage.editor
```

---

## Шаг 2. Создать статический ключ доступа

```bash
yc iam access-key create --service-account-name service-account-for-cf
```

В результате будут получены:

- `key_id` — идентификатор ключа доступа;
- `secret` — секретный ключ.

Их удобно сохранить в переменные окружения:

```bash
export ACCESS_KEY=<key_id>
export SECRET_KEY=<secret>
```

---

## Шаг 3. Создать бакет

В Object Storage нужно создать бакет.

При создании бакета указываются:

- имя бакета;
- максимальный размер;
- тип доступа;
- класс хранилища.

Для учебного сценария можно использовать публичный доступ, но в реальных проектах лучше ограничивать доступ по принципу минимальных прав.

# 📤 Функция загрузки файла в Object Storage

Следующий этап — создать функцию, которая при вызове создаёт временный файл и загружает его в Object Storage.

---

## Зависимости

Для работы с Object Storage используется библиотека `boto3`.

Файл `requirements.txt`:

```text
boto3==1.13.10
botocore==1.16.10
python-dateutil==2.8.1
pytz==2020.1
```

---

## Общая логика функции

Функция выполняет следующие действия:

1. получает переменные окружения;
2. создаёт временный файл в `/tmp`;
3. записывает в него текст;
4. подключается к Object Storage через S3-compatible API;
5. загружает файл в бакет;
6. удаляет временный файл;
7. возвращает HTTP-ответ.

---

## Основные переменные окружения

```text
ACCESS_KEY
SECRET_KEY
BUCKET_NAME
TIME_ZONE
```

---

## Пример структуры функции

```python
import os
import datetime
import boto3
import pytz

ACCESS_KEY = os.getenv("ACCESS_KEY")
SECRET_KEY = os.getenv("SECRET_KEY")
BUCKET_NAME = os.getenv("BUCKET_NAME")
TIME_ZONE = os.getenv("TIME_ZONE", "Europe/Moscow")

TEMP_FILENAME = "/tmp/temp_file"
TEXT_FOR_TEMP_FILE = "This is text file"


def handler(event, context):
    # Создание файла
    # Загрузка файла в Object Storage
    # Удаление временного файла
    return {
        'statusCode': 200,
        'body': 'File is uploaded',
    }
```

---

## Создание ZIP-архива

Так как функция использует зависимости, нужно заархивировать файлы:

```text
index.py
requirements.txt
```

в архив:

```text
my-first-function.zip
```

---

## Загрузка новой версии функции

```bash
yc serverless function version create \
  --function-name my-first-function \
  --memory 256m \
  --execution-timeout 5s \
  --runtime python37 \
  --entrypoint index.handler \
  --service-account-id $SERVICE_ACCOUNT_ID \
  --source-path my-first-function.zip \
  --environment ACCESS_KEY=$ACCESS_KEY \
  --environment SECRET_KEY=$SECRET_KEY \
  --environment BUCKET_NAME=$BUCKET_NAME
```

---

## Проверка функции

```bash
yc serverless function version list --function-name my-first-function
```

Вызов функции:

```bash
yc serverless function invoke <function_id>
```

После вызова в Object Storage должен появиться новый файл.

# 🚀 Создание функции для триггера

Теперь создаётся вторая функция, которую будет запускать триггер.

---

## Назначение функции

Функция `my-trigger-function` должна запускаться после создания нового объекта в бакете и выводить входящее событие `event` в логи.

---

## Пример кода

```python
def handler(event, context):
    print("Starting function after trigger")
    print(event)
    return {
        'statusCode': 200,
        'body': 'File is uploaded',
    }
```

---

## Создание функции

```bash
yc serverless function create --name my-trigger-function
```

---

## Создание версии функции

```bash
yc serverless function version create \
  --function-name my-trigger-function \
  --memory 256m \
  --execution-timeout 5s \
  --runtime python37 \
  --entrypoint index.handler \
  --service-account-id $SERVICE_ACCOUNT_ID \
  --source-path index.py
```

---

# 🔗 Создание триггера Object Storage

Теперь создаётся триггер `my-first-trigger`.

Он будет вызывать функцию `my-trigger-function` при создании нового объекта в бакете.

```bash
yc serverless trigger create object-storage \
  --name my-first-trigger \
  --bucket-id $BUCKET_NAME \
  --events 'create-object' \
  --invoke-function-name my-trigger-function \
  --invoke-function-service-account-id $SERVICE_ACCOUNT_ID
```

# 📌 Что важно запомнить

После лекции важно понимать:

- serverless позволяет запускать код без управления серверами;
- Cloud Functions относятся к модели FaaS;
- функция не хранит состояние между вызовами;
- состояние нужно сохранять во внешнем хранилище;
- функция должна иметь точку входа;
- каждая версия функции фиксирует конкретное состояние кода;
- тег `$latest` указывает на последнюю версию;
- пользовательские теги помогают управлять версиями;
- логи нужны для отладки и контроля выполнения;
- триггеры позволяют запускать функции автоматически;
- Object Storage trigger запускается при событиях в бакете;
- переменные окружения позволяют передавать функции параметры без хардкода;
- функции можно связывать в событийные цепочки.

# 📎 Итог

Лекция 13 показывает, как в Yandex Cloud строятся событийные serverless-сценарии.

Главная идея лекции:

> приложение может состоять не только из серверов и контейнеров, но и из небольших функций, которые автоматически запускаются по событиям и связывают между собой разные облачные сервисы.

Serverless-подход позволяет быстро создавать автоматизацию, интеграции и backend-логику без ручного администрирования инфраструктуры.
