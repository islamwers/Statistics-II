# 🧱 Микросервисная архитектура

## Что такое микросервисная архитектура

**Микросервисная архитектура** — это подход к разработке программного обеспечения, при котором приложение разбивается на небольшие автономные компоненты — микросервисы.

Каждый микросервис:

- отвечает за отдельную бизнес-функцию;
- имеет чётко определённый интерфейс;
- может разрабатываться независимо;
- может масштабироваться отдельно;
- может использовать собственный технологический стек.

---

## Микросервисы vs монолит

| Критерий | Микросервисы | Монолит |
|---|---|---|
| Скорость релизов | Можно обновлять отдельный сервис | Нужно обновлять всё приложение |
| Технологии | Для каждого сервиса можно выбрать свой стек | Вся система обычно использует единый стек |
| Разработка | Команды могут работать параллельно | Новичку нужно понимать всю систему |
| Масштабирование | Масштабируется конкретный сервис | Масштабируется всё приложение |
| Отказоустойчивость | Сбой одного сервиса не всегда ломает всю систему | Ошибка в одном модуле может повлиять на всю систему |
| Оптимизация | Каждый сервис можно оптимизировать отдельно | Нужно учитывать связи между всеми модулями |

---

## Преимущества микросервисов

Микросервисы позволяют:

- быстрее выпускать обновления;
- масштабировать только нагруженные компоненты;
- использовать разные языки программирования и базы данных;
- повысить отказоустойчивость системы;
- организовать параллельную работу разных команд;
- упростить поддержку отдельных модулей.

---

## Ограничения микросервисной архитектуры

Несмотря на преимущества, микросервисы усложняют архитектуру.

Основные сложности:

- необходимость отдельно разворачивать каждый сервис;
- увеличение затрат на тестирование и мониторинг;
- сетевые задержки между сервисами;
- возможная неконсистентность данных;
- сложность координации больших команд;
- необходимость продуманного DevOps-процесса.

---

## Кому подходят микросервисы

Микросервисная архитектура особенно полезна для:

- больших команд разработки;
- объёмных проектов со сложной архитектурой;
- продуктов с резко меняющимся трафиком;
- приложений, требующих частых обновлений;
- систем, где разные команды отвечают за разные части продукта.

---

## Сервисы для работы с микросервисами

Для построения микросервисной инфраструктуры используются:

- **Docker** — платформа контейнеризации;
- **Kubernetes** — оркестратор контейнеров;
- **ALB / NLB** — балансировщики нагрузки для распределения трафика.

# 📦 Контейнеризация

## Что такое контейнеризация

**Контейнеризация** — это подход к разработке и развёртыванию программного обеспечения, при котором приложение, его зависимости, библиотеки и конфигурация упаковываются в единый контейнерный образ.

Контейнер позволяет запускать приложение одинаково в разных средах:

- на локальном компьютере;
- на сервере;
- в облаке;
- в Kubernetes-кластере.

---

## Контейнеризация и виртуализация

Контейнеризация отличается от классической виртуализации тем, что контейнеры используют ядро хостовой операционной системы и не требуют отдельной гостевой ОС для каждого приложения.

Это делает контейнеры:

- более лёгкими;
- более быстрыми;
- удобными для масштабирования;
- эффективными с точки зрения ресурсов.

---

## Преимущества контейнеров

Контейнеры дают несколько важных преимуществ.

### 1. Экономия ресурсов

Контейнеры не содержат отдельную копию операционной системы, поэтому занимают меньше места и быстрее запускаются.

### 2. Независимость

Контейнер содержит всё необходимое для запуска приложения:

- код;
- библиотеки;
- зависимости;
- настройки;
- среду выполнения.

### 3. Переносимость

Контейнер можно запускать на разных платформах без изменения приложения.

### 4. Скорость развёртывания

Контейнеры запускаются быстрее виртуальных машин.

### 5. Тиражирование и масштабирование

Один и тот же контейнер можно запустить в нескольких копиях.

### 6. Оркестрация

Контейнеры удобно объединять в большие системы и управлять ими через Kubernetes или Docker Swarm.

---

# 🐳 Docker

## Что такое Docker

**Docker** — это платформа контейнеризации с открытым исходным кодом, которая позволяет автоматизировать создание, доставку и управление приложениями.

Docker помогает разработчикам собирать приложение вместе со всеми зависимостями и запускать его в любой совместимой инфраструктуре.

---

## Преимущества Docker

Docker позволяет:

- быстро разворачивать приложения;
- уменьшать количество ошибок из-за различий окружений;
- переносить приложения между средами;
- использовать любые языки программирования и технологии;
- переиспользовать готовые образы;
- упрощать администрирование;
- повышать безопасность за счёт изоляции;
- строить микросервисные приложения.

---

## Основные компоненты Docker

| Компонент | Описание |
|---|---|
| Dockerfile | Текстовый файл с инструкциями для сборки образа |
| Docker Daemon | Фоновая служба, управляющая контейнерами |
| Docker Image | Неизменяемый образ, из которого создаются контейнеры |
| Docker Client | CLI-утилита для управления Docker |
| Container | Запущенный экземпляр образа |
| Registry | Хранилище Docker-образов |
| Docker Hub | Публичный реестр Docker-образов |
| Docker Desktop | Приложение для локальной работы с Docker |
| Docker Volumes | Механизм постоянного хранения данных |

---

# 🧬 Устройство Docker-образа

Docker-образ состоит из слоёв.

Основные идеи:

- базовый образ задаёт начальную среду;
- каждая инструкция Dockerfile создаёт новый слой;
- слои доступны только для чтения;
- контейнер добавляет поверх образа временный слой с возможностью записи;
- при удалении контейнера временный слой исчезает;
- образ можно использовать многократно.

---

# 📝 Dockerfile

## Пример Dockerfile

```dockerfile
FROM ubuntu:latest
RUN apt-get update -y
RUN apt-get install -y nginx
ENTRYPOINT ["nginx", "-g", "daemon off;"]
```

Разбор:

- `FROM ubuntu:latest` — выбирает базовый образ Ubuntu;
- `RUN apt-get update -y` — обновляет список пакетов;
- `RUN apt-get install -y nginx` — устанавливает NGINX;
- `ENTRYPOINT ...` — запускает NGINX при старте контейнера.

---

## Сборка Docker-образа

Если файл называется `Dockerfile`, можно использовать команду:

```bash
docker image build -t my-image .
```

Если Dockerfile имеет другое имя:

```bash
docker image build -f my-dockerfile -t my-image .
```

Точка в конце означает, что контекст сборки — текущая директория.

---

## Работа с образами

Загрузить образ в реестр:

```bash
docker push my-image
```

Скачать образ из реестра:

```bash
docker pull my-image
```

Запустить контейнер из образа:

```bash
docker run my-image
```

# 🟡 Yandex Container Registry

## Что такое Yandex Container Registry

**Yandex Container Registry** — это управляемый сервис Yandex Cloud для хранения и распространения Docker-образов.

Он позволяет хранить образы приложений, управлять доступом к ним и использовать их при развёртывании контейнеров в облаке.

---

## Преимущества Yandex Container Registry

### 1. Приватный реестр по умолчанию

В отличие от публичных Docker-реестров, реестр в Yandex Cloud по умолчанию является приватным.

### 2. Бесплатный внутренний трафик

Если образы используются внутри Yandex Cloud, трафик считается внутренним.

### 3. Удобство

Сервис доступен через консоль управления и CLI `yc`.

### 4. Политики автоматического удаления

Можно настроить правила очистки старых или тестовых образов.

---

## Реестр, репозиторий и тег

В Yandex Container Registry используются три ключевых понятия:

- **реестр** — хранилище Docker-образов;
- **репозиторий** — набор образов с одинаковым именем;
- **тег** — версия образа.

Формат полного имени образа:

```text
cr.yandex/<реестр>/<имя_образа>:<тег>
```

Пример:

```text
cr.yandex/my-registry/my-app:latest
```

Если тег не указан, используется тег:

```text
latest
```

---

# 🧹 Политики автоматического удаления образов

Политики удаления позволяют автоматически очищать старые Docker-образы.

Основные параметры:

- `tag_regexp` — фильтр по тегу образа;
- `untagged` — правило для образов без тегов;
- `expire_period` — срок хранения;
- `retained_top` — сколько последних образов нужно сохранить.

Пример JSON-файла политики:

```json
[
  {
    "description": "Delete prod Docker images older than 30 days but retain 20 last ones",
    "tag_regexp": "prod",
    "expire_period": "30d",
    "retained_top": 20
  },
  {
    "description": "delete all test Docker images except 10 last ones",
    "tag_regexp": "test.*",
    "retained_top": 10
  },
  {
    "description": "delete all untagged Docker images older than 48 hours",
    "untagged": true,
    "expire_period": "48h"
  }
]
```

Перед применением политики рекомендуется выполнить тестовый запуск:

```text
dry-run
```

# 🖥️ Container Optimized Image

**Container Optimized Image** — это образ виртуальной машины, оптимизированный для запуска Docker-контейнеров.

Он включает:

- Ubuntu LTS;
- Docker;
- демон для запуска Docker-контейнеров.

Преимущества:

- Docker-контейнер можно запустить сразу после создания ВМ;
- можно обновлять контейнер с минимальным простоем;
- открытые сетевые порты контейнера доступны без дополнительных настроек.

---

# 🚀 Практическая часть: Docker-образ и Container Registry

В практической части создаётся Docker-образ с NGINX, загружается в Yandex Container Registry и запускается через Container Optimized Image.

---

## Шаг 1. Установить Docker

Docker устанавливается по официальной инструкции для вашей операционной системы.

---

## Шаг 2. Создать реестр

```bash
yc container registry create --name my-registry
```

В выводе команды нужно сохранить уникальный идентификатор реестра.

---

## Шаг 3. Настроить аутентификацию Docker

```bash
yc container registry configure-docker
```

Эта команда позволяет Docker отправлять образы в приватный реестр Yandex Cloud.

---

## Шаг 4. Подготовить Dockerfile

```dockerfile
FROM ubuntu:latest
RUN apt-get update -y
RUN apt-get install -y nginx
ENTRYPOINT ["nginx", "-g", "daemon off;"]
```

---

## Шаг 5. Собрать Docker-образ

```bash
docker build . -t cr.yandex/<идентификатор_реестра>/ubuntu-nginx:latest
```

---

## Шаг 6. Загрузить образ в реестр

```bash
docker push cr.yandex/<идентификатор_реестра>/ubuntu-nginx:latest
```

---

## Шаг 7. Настроить права доступа

Для запуска образа из реестра нужно выдать права:

- `viewer`;
- `container-registry.images.puller`.

В лекции рассматривается вариант выдачи прав группе `All users`.

---

## Шаг 8. Создать ВМ через Container Optimized Image

При создании ВМ нужно:

1. Выбрать образ загрузочного диска.
2. Перейти на вкладку **Container Solution**.
3. Нажать **Настроить**.
4. Выбрать Docker-образ из реестра.
5. Завершить создание ВМ.

После запуска ВМ можно открыть внешний IP-адрес и увидеть приветственную страницу NGINX.

# 🤖 MLOps

## Что такое MLOps

**MLOps** — это инженерная дисциплина, которая объединяет разработку, развёртывание и эксплуатацию систем машинного обучения.

Цель MLOps — стандартизировать и оптимизировать доставку ML-моделей в промышленную среду.

MLOps находится на пересечении:

- Machine Learning;
- DevOps;
- Data Engineering.

---

## Какие вопросы решает MLOps

MLOps помогает ответить на вопросы:

1. Как организовать процесс разработки ML-моделей?
2. Как выстроить сквозной жизненный цикл моделей?
3. Как организовать взаимодействие между командами?
4. Как управлять версиями моделей?
5. Как сократить время доставки моделей в production?
6. Что делать после внедрения модели?

---

## Основные принципы MLOps

Ключевые принципы:

- быстрое развёртывание моделей в рабочей среде;
- ускорение разработки экспериментальных моделей;
- контроль качества;
- сквозное отслеживание происхождения данных, кода и модели;
- автоматизация ML-pipeline;
- мониторинг моделей после внедрения.

---

## Уровни зрелости MLOps

В лекции рассматриваются уровни зрелости MLOps.

### MLOps Level 0

Ручной процесс:

- эксперименты выполняются вручную;
- слабая автоматизация;
- воспроизводимость ограничена.

### MLOps Level 1

Автоматизация ML-pipeline:

- появляется структурированный pipeline;
- часть процессов автоматизирована;
- результаты можно воспроизводить.

### MLOps Level 2

Автоматизация CI/CD pipeline:

- автоматическая сборка;
- автоматическое тестирование;
- автоматическая доставка моделей;
- полноценный производственный контур.

---

## Типовая роль MLOps-инженера

MLOps-инженер может работать со стеком:

- Kubernetes;
- Docker;
- Airflow;
- MLflow;
- Python;
- Spark;
- Hadoop;
- CI/CD;
- Linux;
- Kubeflow;
- Helm;
- Terraform.

Типовые задачи:

- поддержка ML-пайплайнов;
- автоматизация обучения моделей;
- деплой моделей;
- мониторинг качества моделей;
- сопровождение production-инфраструктуры;
- управление версиями моделей и данных.
