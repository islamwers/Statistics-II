# ⚖️ OLTP и OLAP

## OLTP

**OLTP** — Online Transaction Processing.

OLTP-системы предназначены для обработки большого количества транзакций в режиме онлайн.

Основные признаки OLTP:

- большое количество операций;
- быстрый отклик, обычно в миллисекундах;
- данные хорошо структурированы;
- используется много таблиц;
- резервное копирование обязательно;
- отвечает на вопрос: **«Кто купил X?»**

Примеры OLTP-сценариев:

- онлайн-банкинг;
- бронирование авиабилетов;
- отправка сообщений;
- добавление товара в корзину;
- обработка заказов.

---

## OLAP

**OLAP** — Online Analytical Processing.

OLAP-системы предназначены для быстрого анализа больших объёмов данных.

Основные признаки OLAP:

- большие объёмы данных;
- отклик обычно измеряется секундами;
- данные могут быть слабо упорядочены;
- таблиц обычно меньше;
- резервное копирование может быть опциональным;
- отвечает на вопрос: **«Как много людей купили X?»**

Примеры OLAP-сценариев:

- анализ продаж;
- анализ посещаемости сайта;
- продуктовая аналитика;
- BI-отчётность;
- анализ поведения пользователей.

# 🗄️ Размещение баз данных

В лекции сравниваются два подхода к размещению БД:

1. самостоятельная установка на виртуальной машине;
2. использование управляемых сервисов Yandex Cloud.

---

## БД на виртуальной машине

При самостоятельном размещении БД на ВМ нужно:

1. создать виртуальную машину;
2. установить БД и необходимые программы;
3. создать конфигурационные файлы;
4. настроить сеть;
5. настроить репликацию, если ВМ несколько;
6. организовать резервное копирование;
7. устанавливать обновления ОС и СУБД;
8. самостоятельно восстанавливать сервис при сбоях.

Плюсы:

- полный контроль над конфигурацией;
- потенциальная экономия на managed-сервисах.

Минусы:

- много ручной работы;
- ответственность за администрирование;
- необходимость самостоятельно настраивать отказоустойчивость и резервное копирование.

---

## Managed-сервисы

Managed-сервисы позволяют создать БД через:

- консоль управления;
- CLI;
- API.

Плюсы:

- меньше ручного администрирования;
- можно сосредоточиться на приложении;
- проще обновлять и поддерживать БД;
- облако берёт на себя часть инфраструктурных задач.

Минусы:

- обычно дороже, чем самостоятельная установка на ВМ.

---

# 📌 Базовые понятия managed-БД

## Кластер

**Кластер** — это одна или несколько виртуальных машин или хостов, где разворачивается база данных.

---

## Окружение

Окружение может быть:

- `PRESTABLE`
- `PRODUCTION`

### PRESTABLE

Используется для разработки и тестирования.  
В него раньше попадают обновления СУБД и сервиса, но они могут содержать ошибки или быть несовместимыми.

### PRODUCTION

Используется для стабильной рабочей версии приложения.  
В него попадают проверенные обновления.

---

## Масштабирование БД

Существует два типа масштабирования:

### Вертикальное масштабирование

Меняется мощность хоста:

- количество ядер;
- объём памяти;
- объём диска.

### Горизонтальное масштабирование

Меняется количество хостов, между которыми распределяется нагрузка.

---

# 🧱 Реляционные базы данных

## Что такое реляционная БД

**Реляционная база данных** — это набор таблиц и связей между ними.

Данные в реляционной модели:

- имеют строгую структуру;
- организованы в таблицы;
- связаны между собой;
- обрабатываются с помощью SQL.

Для работы с реляционными БД используется:

```text
SQL — Structured Query Language
```

---

## Основные реляционные БД в Yandex Cloud

### PostgreSQL

Объектно-реляционная СУБД с открытым исходным кодом.

Особенности:

- высокая надёжность;
- масштабируемость;
- подходит для финансового учёта;
- часто используется в backend-сервисах.

---

### MySQL

Популярная свободная СУБД.

Особенности:

- гибкость;
- простота;
- удобство использования;
- широкая распространённость.

---

### Greenplum

Аналитическая колоночная MPP-СУБД.

Особенности:

- подходит для сложной аналитики;
- работает с большими объёмами данных;
- поддерживает ANSI SQL;
- поддерживает joins;
- масштабируется на десятки терабайт.

---

## Плюсы реляционных БД

- данные структурированы;
- сохраняется целостность данных;
- популярность и зрелость технологий;
- поддержка SQL;
- удобство для транзакционных систем.

---

## Минусы реляционных БД

- хуже работают при смешанной нагрузке чтения и записи;
- сложнее масштабируются;
- не все данные удобно укладываются в строгую табличную структуру;
- изменение схемы может быть трудозатратным.

---

# 📄 Нереляционные базы данных

## Что такое нереляционная БД

**Нереляционная база данных** — это БД, в которой не используется классическая табличная схема строк и столбцов.

В таких БД данные могут храниться как:

- пары `ключ — значение`;
- JSON-документы;
- графы;
- колонки;
- события и потоки данных.

---

## Основные нереляционные БД в Yandex Cloud

### MongoDB

Документоориентированная NoSQL-СУБД.

Особенности:

- данные хранятся в JSON-подобных документах;
- документы объединяются в коллекции;
- удобно хранить данные с разным набором полей;
- масштабируется за счёт шардирования.

---

### Redis

Хранилище типа `key-value`.

Особенности:

- данные хранятся в оперативной памяти;
- очень высокая производительность;
- подходит для миллионов операций в секунду;
- запросы могут использовать Lua-скрипты.

Используется для:

- хранения сессий;
- кеширования;
- таблиц результатов;
- игровых и веб-приложений.

---

### ClickHouse

Колоночная СУБД для онлайн-аналитики.

Особенности:

- хранение данных по столбцам;
- чтение сотен миллионов записей в секунду;
- горизонтальное масштабирование;
- работа с базами размером до петабайт.

---

### Elasticsearch / OpenSearch

Нереляционное хранилище с полнотекстовым поиском.

Особенности:

- данные хранятся в JSON-документах;
- быстрый поиск;
- масштабируемость;
- аналитика больших массивов данных.

---

### YDB

Распределённая СУБД, разработанная Яндексом.

Особенности:

- относится к NewSQL;
- основана на реляционной модели;
- обеспечивает отказоустойчивость;
- обеспечивает доступность;
- масштабируется как NoSQL-решения;
- поддерживает транзакции;
- использует SQL-подобный язык запросов.

---

### Apache Kafka

Apache Kafka — распределённая стриминговая платформа.

Она сочетает свойства:

- очереди сообщений;
- распределённого хранилища событий;
- системы передачи потоков данных.

Kafka используется для надёжной передачи большого потока сообщений между сервисами.

---

## Когда использовать нереляционные БД

Нереляционные БД подходят, если:

- нужно хранить большие объёмы неструктурированных данных;
- схема данных заранее неизвестна;
- структура данных может часто меняться;
- нужно быстро запустить прототип;
- требуется высокая масштабируемость;
- строгой согласованностью можно пожертвовать ради производительности и доступности.

При выборе базы данных важно учитывать:

- тип данных;

# 🪣 Object Storage

## Что такое Object Storage

**Object Storage** — это облачная система хранения неструктурированных данных произвольного формата.

В Object Storage можно хранить:

- файлы;
- изображения;
- видео;
- архивы;
- резервные копии;
- логи;
- датасеты;
- статические сайты.

---

## Объект

В хранилище записываются **объекты**.

Объект — это аналог файла.

У каждого объекта есть:

- уникальный идентификатор;
- метаданные;
- дата создания;
- MD5-хеш;
- другие служебные сведения.

---

## Бакет

Объекты группируются в **бакеты**.

Бакеты позволяют разделять данные:

- разных проектов;
- разных пользователей;
- разных приложений;
- разных окружений.

Важно:

```text
имя бакета в Yandex Cloud должно быть уникальным
```

---

## Преимущества Object Storage

Object Storage даёт следующие преимущества:

- масштабируемость;
- отсутствие архитектурного ограничения на объём данных;
- высокая производительность при работе с большими объёмами;
- отказоустойчивость;
- хранение копий данных в географически распределённых дата-центрах;
- поддержка метаданных;
- низкая стоимость хранения.

---

## Особенности Object Storage

Object Storage рассчитан в первую очередь на взаимодействие через:

```text
HTTP API
```

Особенности:

- объекты не редактируются напрямую;
- для изменения объекта нужно загрузить новую версию;
- не всегда удобно для часто изменяемых больших файлов;
- лучше подходит для хранения, чем для постоянного редактирования.

# 🔐 Управление доступом

В Object Storage используются три независимых механизма управления доступом.

---

## 1. IAM

**IAM (Identity and Access Management)** работает на уровне облака.

Он управляет правами пользователей, сервисных аккаунтов и групп.

---

## 2. ACL

**ACL (Access Control List)** работает на уровне бакетов и объектов.

ACL может задавать права:

- `READ` — чтение объектов и бакетов;
- `WRITE` — запись, перезапись и удаление объектов;
- `FULL_CONTROL` — полный доступ.

---

## 3. Bucket Policy

**Bucket Policy** задаёт условия доступа к бакетам и объектам.

Элементы политики:

- ресурс;
- действие;
- результат;
- пользователь;
- условие;
- принцип выбора пользователей.

---

# 🔄 Жизненный цикл объектов

Жизненный цикл объектов позволяет оптимизировать затраты на хранение.

Можно настроить правила:

- перехода объекта в другой класс хранения;
- автоматического удаления объекта;
- обработки объектов по дате;
- обработки объектов через заданное количество дней после загрузки.

---

## Типы операций

### Transition

Перевод объекта в другой класс хранения.

Например:

```text
Стандартное → Холодное → Ледяное
```

### Expiration

Автоматическое удаление объекта.

---

# 🧬 Версионирование

**Версионирование** позволяет сохранять историю объекта.

Если версионирование включено:

- объект не теряется после удаления или перезаписи;
- можно получить доступ к предыдущим версиям;
- каждый объект получает `version_id`;
- версионирование применяется ко всем объектам внутри бакета.

Версионирование полезно для:

- защиты от случайного удаления;
- восстановления предыдущих версий;
- хранения истории изменений.

# 🌐 Хостинг статических сайтов в Object Storage

Object Storage можно использовать для размещения полностью статических сайтов.

Подходит для сайтов на:

- HTML;
- CSS;
- JavaScript.

Не подходит для сайтов, где требуется серверная логика.

---

## Основные шаги

1. Создать бакет.
2. Если используется собственный домен, назвать бакет так же, как домен.
3. Открыть публичный доступ на чтение объектов.
4. Загрузить файлы сайта.
5. Включить хостинг во вкладке **Веб-сайт**.
6. Указать главный файл, обычно:

```text
index.html
```

7. Сохранить настройки.

---

## Адреса сайта

После настройки сайт будет доступен по адресам:

```text
http://<имя_бакета>.website.yandexcloud.net
```

или

```text
http://website.yandexcloud.net/<имя_бакета>
```

---

## HTTPS и домен

По умолчанию сайт доступен по HTTP.

Для HTTPS необходимо загрузить собственный сертификат безопасности.

Для собственного домена можно настроить CNAME-запись, например:

```text
www.example.com CNAME www.example.com.website.yandexcloud.net
```

Важно:

```text
можно использовать домены не ниже третьего уровня
```

Например, `www.example.com` использовать можно, а `example.com` — нет.

# 📝 Практическая часть лекции

В практической части студент выполняет следующие действия:

1. Изучает сервисы платформы данных.
2. Разбирает различия между OLTP и OLAP.
3. Сравнивает managed-БД и БД на ВМ.
4. Изучает реляционные и нереляционные СУБД.
5. Разбирает схему выбора БД.
6. Открывает Object Storage.
7. Создаёт бакет.
8. Загружает объект в бакет.
9. Получает ссылку на объект.
10. Настраивает жизненный цикл объектов.
11. Изучает версионирование.
12. Настраивает хостинг статического сайта.

---

# 📌 Что важно запомнить

После лекции важно понимать:

- OLTP используется для транзакционных систем;
- OLAP используется для аналитики больших данных;
- managed-БД позволяют снизить нагрузку на администрирование;
- реляционные БД подходят для структурированных данных и транзакций;
- нереляционные БД подходят для гибких, масштабируемых и неструктурированных данных;
- Object Storage используется для хранения файлов и неструктурированных данных;
- бакет — это контейнер для объектов;
- доступ можно управлять через IAM, ACL и Bucket Policy;
- жизненный цикл помогает оптимизировать стоимость хранения;
- версионирование защищает от потери данных;
- Object Storage можно использовать для статического сайта.
